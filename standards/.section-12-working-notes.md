## Document Purpose (Agreed)

Section 12 exists to establish a **consistent monitoring and alerting baseline** that makes the Data Platform governable, auditable, and safe to operate at scale.

The root cause it addresses is **inconsistency** — which cascades into governance gaps, which enables silent failures.

### Primary Failure Modes (explicitly addressed)
1. **Silent failures** — pipelines fail or produce bad data with no detection
2. **Drift without detection** — pipelines technically succeed but data meaning quietly degrades (sum mismatches, late arrivals, quality drift). No alert fires because nothing "broke". Named separately from silent failures because it requires different detection strategies.
3. **Governance gaps** — no audit trail to prove monitoring was in place
4. **Inconsistency** — every product team monitors differently, making platform-wide visibility impossible (root cause of the above)

### Intended Readers & Core Needs

| Role | Core Need |
|---|---|
| **Project Developer** | What must I instrument before my code ships, and how do I make late data / quality drift / sum mismatches detectable without building it from scratch each time? |
| **Platform Engineer** | What's the minimum viable monitoring baseline that gives platform-wide visibility without requiring me to understand every product team's domain? |
| **Governance Reviewer** | Can I prove — not just assume — that this pipeline's data is compliant? (downstream consumption visibility / Purview → deferred to separate governance section) |
| **Incident Responder** | Is the alert telling me what broke, where, and what the blast radius is — or am I starting from zero? |

### Scope Decisions
- **MS Purview** (lineage, downstream consumption visibility) → deferred to a separate governance section
- **"Drift without detection"** → named as a distinct failure mode in the document purpose statement

## Terminology (Agreed)

```
Business Domain          (e.g. Transport Safety)
  └── Data Product       (BAU, long-lived, owned by a domain team)
        └── Project      (short-lived, delivers an increment to the data product)
              └── Pipelines  (physical — ADF pipelines, Databricks jobs)
```

- "Product team" in StatsNZ source docs → means **domain team** at NZTA
- "Project team" → short-lived delivery team; pipelines transfer to domain team at completion
- **ITSM tool** → ServiceNow (not IvantiCloud — inherited from StatsNZ, not applicable at NZTA)
- **Environments** → Dev, Test, Prod only (no UAT at NZTA)

## Subsection Decisions

### 12.1 Monitoring Scope

**Ownership model (two-tier):**
- Platform team → accountable for infrastructure + platform service monitoring AND for providing shared monitoring frameworks/tooling that domain/project teams use
- Domain team → accountable for pipeline and data product monitoring WITHIN the frameworks the platform team provides
- The seam between layers (e.g. ADF triggering Databricks) → platform team responsibility
- Monitoring ownership must be formally transferred from project team to domain team upon project completion

**Environment monitoring requirements:**

| Environment | Requirement |
|---|---|
| Dev | Logging must be present; alerting optional |
| Test | Logging required; alerting on pipeline failures required |
| Prod | Full monitoring; all critical alerts active and routed to ServiceNow |

**Dropped:** "higher rigour in UAT and Production" — too vague, replaced with explicit per-environment requirements above
